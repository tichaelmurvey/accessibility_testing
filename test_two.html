List tagging and screen reader interactions test

Test 1: Auto-tagged list

List was created using the text tool in Acrobat, then tagged with the autotag feature in Acrobat.

View the testing document for test 1[PDF].

NVDA + Acrobat

Announced list with 4 items. On “I” moves through list body text, skipping the first item.

NVDA + Firefox

Announced list with 4 items. On “I” moves through list body text, skipping the first item.

NVDA + Chrome

List content announced as plain text. Text was not identified as a list, and list shortcuts returned “No next list”.

JAWS + Acrobat

Announced list with 4 items. On “I” moves through bullets (tagged as label). Arrow key navigation required to access list body.

JAWS + Firefox

Announced list with 4 items. On “I” moves through list items, focusing bullet and text at the same time. List items are announced as “Bullet [body text] graphic”.

JAWS + Chrome

Did not announce list. On L/I shortcuts, announces no lists or list items.

List items (including bullet and text) can be focused with arrow keys and are announced as “Bullet [body text]”.

View the testing document for test 2[PDF].

Deleted a single list item tag, and recreated the tag structure using “Create tag from content” and “New tag”.

NVDA + Acrobat

The recreated list item was ignored by NVDA. The list was announced as 3 items rather than 4.

Announced as list with 4 items. List navigation and shortcuts functioned as expected.

JAWS + Firefox

JAWS + Chrome

Findings

  1. Lists in PDFs are not supported by Chrome.
  2. Firefox is the only user agent which announced the intended list markup across all tests.
  3. JAWS does not seem to have a great relationship with the Label element. On Acrobat it is assumed to be the link element.
  4. In cases where list item tags are deleted and re-created, Acrobat will not announce those list items to screen readers. This creates unpredictable behaviour in JAWS and NVDA. I have investigated the content tree and tag properties for the newly created tag. I cannot find any differences between the two constructions. Therefore, I’m calling this an Acrobat bug.
  5. The bug identified in item 4 does not occur when the tag tree is deleted, and new tags are created from content.

Speculation

The “Graphic” announcement of JAWS on Firefox may be recognising the bullets. This would be consistent with JAWS not recognising label tags on Acrobat.

Further testing

Most importantly, tests should be done to determine which if any other elements are affected by the delete + retag bug on Acrobat. I have tested the same process with a heading element, and the bug was not reproduced.

Tests should be done to determine behaviour when elements are incorrectly tagged at initial markup, then retagged as list items later. For example, if an item is initially tagged as a <P> then retagged as a list item, does the Acrobat bug apply?

Test should be done to determine if the Acrobat bug occurs when new content is added via the edit tool, then manually tagged. If the explanation of the results above is correct, this content should be announced as it has not previously been tagged. However, this has not been tested and the results may be unexpected.

I would like to know how screen readers react to tagging bullets as artefacts rather than labels. This may improve quality of content flow on JAWS, but may also break the list flow.
